# Appinn 发帖发布

## 目标

将工具介绍整理为可发布到小众软件论坛的帖子，并尽量提高通过率与可读性。

## 固定规则（必须满足）

1. 先按规则自检：<https://meta.appinn.net/t/topic/43728>
2. 标题不要出现“免费”字样，除非全流程实测确实免费；如测试中出现付费，视为不合规。
3. 禁止标题党：标题和正文必须真实、克制、信息一致。
4. 需要注册才能使用时，必须在正文明显位置说明。
5. 必须保留官网链接。
6. 已发布内容应直接编辑，不要删除后重发。
7. 帖子语言尽量简洁，排版整洁，优先“易读、可信、可验证”。
8. 必须严格按照排版范例进行版式组织与信息分段：<https://meta.appinn.net/t/topic/23842>（不允许仅套模板而忽略范例的段落节奏与可读性要求）。

## 输出流程

按以下顺序输出内容：

1. 先确认发帖角色：`开发者自荐` 或 `用户推荐`（若未说明必须先问）。
2. 按角色执行「信息采集提问」并收集素材（信息不足时先追问，不直接生成完整帖子）。
3. 再给「合规检查结果」：逐条列出通过/待补充。
4. 再给「可直接发布的帖子正文」。
5. 最后给「发布前 30 秒复核清单」。

## 信息采集提问（按角色）

在生成帖子前，必须先问问题收集信息；不同角色使用不同问题集。

### 角色 A：开发者自荐

目标：帮助开发者把“产品价值、解决的问题、真实门槛”讲清楚，而不是营销化堆词。

必问问题（至少覆盖以下 8 项）：

1. 你的工具主要帮用户解决什么具体问题？（一句话说清“谁在什么场景下卡住了什么”）
2. 相比现有替代方案（同类软件/手工流程），你的优势是什么？请给 2~3 条可验证差异。
3. 最核心的 3 个功能分别是什么？每个功能解决什么痛点？
4. 目标用户是谁？最典型的使用场景是什么？（请给 1~2 个真实场景）
5. 使用门槛是什么？是否需要注册、邀请码、联网、地区限制？
6. 付费方式与触发点是什么？（免费/订阅/买断；从哪个功能开始收费）
7. 当前已知不足或限制有哪些？（例如性能、平台覆盖、功能边界）
8. 你最希望读者先体验哪一步？（给出一个 30 秒可上手的入口）

可选追问（信息不足时）：

- 是否有官网、文档、演示视频、更新日志可用于核验？
- 是否有不适合人群或不推荐使用的场景？

### 角色 B：用户推荐

目标：突出真实使用体验，说明“为什么值得用”以及“哪里还不完美”。

必问问题（至少覆盖以下 8 项）：

1. 你是为了解决什么实际问题开始使用它的？
2. 你已经使用了多久、使用频率如何？（每天/每周/偶尔）
3. 你最常用的 2~3 个功能是什么？各自带来的具体收益是什么？
4. 和你之前用过的方案相比，它最打动你的点是什么？
5. 在真实使用中遇到过哪些不足、坑点或限制？
6. 上手门槛如何？是否需要注册、付费、科学上网或额外配置？
7. 你认为它最适合哪些人，不适合哪些人？
8. 如果只给新用户一条建议，你会建议他先做什么？

可选追问（信息不足时）：

- 是否有可复现的使用案例（任务类型、耗时变化、效果对比）？
- 你目前是否仍持续使用？若停用，主要原因是什么？

## 帖子模板（默认结构）

按下面结构生成，不要省略“注册/付费说明”：

```markdown
# {工具名}（{一句定位，避免夸张词}）

{一句话简介：这是什么，解决什么问题}

## 软件名称
{中文名 / 英文名}

## 官网
{官网 URL}

## 平台
- Windows
- macOS
- Linux
- Web
- iOS / Android

## 推荐类型
【开发者自荐】/【编辑推荐候选】/【体验分享】

## 使用门槛（必写）
- 是否需要注册：{需要 / 不需要}
- 是否付费：{免费 / 部分付费 / 订阅制 / 买断}
- 付费触发点：{例如导出高清、批量处理、云同步等；无则写“未发现”}

## 一句话简介
{20~40 字，避免营销话术}

## 应用简介
1. {核心功能 1}
2. {核心功能 2}
3. {核心功能 3}
4. {差异点或亮点}

## 适用人群
- {人群 A}
- {人群 B}

## 已知限制（建议写）
- {限制 1}
- {限制 2}

## 演示内容（可选）
{如需嵌入视频，将视频链接单独放一行；支持 mp4/Bilibili/YouTube}
```

## 写作约束

- **零编造原则**：只使用我提供的资料中的信息，绝不编造功能，用户反馈，产品介绍等。
- 使用陈述句，不用“最强、神器、必备、吊打”等词。
- 每段尽量短句，列表优先，减少大段堆叠文本。
- **禁止猜测** 未实际验证的信息标记为“待确认”，不要猜测没有提供给你的信息。
- 如果产品需要登录、邀请码、地区限制，必须前置说明。
- 默认使用白话表达，优先普通用户能看懂的词汇；避免专业术语、英文缩写与行话。
- 若必须出现专业词，紧跟一句通俗解释（例如“API Key：调用服务时用的密钥”）。
- 严禁代替用户编造体验：对于“用户推荐”场景，只能基于提问获得的真实反馈组织文案。
- 对于“开发者自荐”场景，优先呈现产品价值与适用边界，不写空泛口号。

## 排版执行规范（强制）

- 以排版范例为基准组织内容顺序、标题层级、段落长度与留白节奏：<https://meta.appinn.net/t/topic/23842>。
- 标题、一级标题、二级标题之间需有清晰分隔；避免连续大段文字，优先短段 + 列表。
- “使用门槛（注册/付费/触发点）”必须前置且醒目，不得埋在文末。
- 链接（官网、视频）单独成行，避免与说明句混排导致可读性下降。
- 输出前必须自检：若与排版范例风格冲突，先重排再输出，不直接提交。

- [ ] 正文明确写出付费模式与触发点
- [ ] 官网链接可访问

- [ ] 全文用词通俗，小白可读；如有专业词，已补一句解释
